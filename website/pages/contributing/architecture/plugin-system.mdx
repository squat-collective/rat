# Plugin System

RAT's plugin system lets you extend the platform without touching core code. The design rests on a few ideas:

- The core ships **no-op defaults** for everything optional, so RAT runs fully standalone with nothing installed.
- A **plugin** is *any* process that speaks RAT's plugin contract. It is not part of the platform — it can be written in any language, live in any repo, and ship on its own schedule.
- Plugins are **self-describing**. A plugin **phones home** to ratd, ratd asks it to `Describe` itself, and the plugin reports what it offers: capabilities, HTTP routes, event subscriptions, a UI bundle, a config schema. ratd wires up only what the plugin advertises.
- ratd keeps an **open registry** of live plugins and acts as the front door: it reverse-proxies plugin HTTP, runs their health checks, fans out events to them, stores their config, and serves their UI to the portal.

So the platform doesn't ship "the auth feature" or "the cloud feature" — it ships a **contract and a registry**. What plugs in is up to you.

---

## Two Layers

"Plugin" means one of two different things, loaded by different mechanisms. Don't conflate them:

| Layer | Shape | Loaded by | Extends |
|-------|-------|-----------|---------|
| **Runner plugin** | A Python package (no Dockerfile) | `pip` + Python `entry_points`, discovered per pipeline run | The **data layer** — merge strategies, hooks, jinja helpers, pipeline types, sources |
| **Platform plugin** | A container that speaks gRPC (and usually HTTP) | **Phones home** to ratd; ratd reverse-proxies it at `/api/v1/x/<name>/*` | The **platform** — new APIs, background workers, auth, events, and portal UI |

When people say "a RAT plugin" they usually mean the **platform** kind. Most of this page is about those; [runner plugins](#runner-plugins-layer-1) are covered at the end.

## How a Platform Plugin Works

A platform plugin is a long-running container. It registers itself with ratd at startup and from then on ratd treats it as part of the running system.

<Steps>

### Phone home

On startup the plugin POSTs its name and address to ratd's **internal listener** — `POST /api/v1/internal/plugins/register`. (The internal listener is unauthenticated by design; trust is network isolation — see [services](/contributing/architecture/services).) The plugin retries with backoff until ratd answers, so start order doesn't matter.

### Describe

ratd calls the plugin's `Describe` RPC and gets back everything it needs to wire the plugin in:

| Field | What it does |
|-------|--------------|
| `capabilities` | Well-known roles the plugin fills (see below) |
| `routes` | HTTP routes ratd should expose and proxy |
| `event_subscriptions` | Platform events the plugin wants delivered |
| `ui` | A portal bundle URL, nav items, UI slots, client routes |
| `config_schema_json` | JSON Schema for the plugin's settings |
| `provides_worker` | Whether the plugin runs background work |
| `platform_token` | A per-startup secret ratd echoes back on every proxied request |

### Record in the registry

ratd stores the plugin in its in-memory **registry** (indexed by capability) and persists it in the `plugin_catalog` Postgres table, so registrations survive restarts.

### Wire it in

From the `Describe` response, ratd starts proxying the plugin's routes, subscribes it to the events it asked for, exposes its UI to the portal, and begins health-checking it.

</Steps>

<Callout type="info">
Plugins **self-register** — there's no central list of plugins to edit. (A legacy path still lets you declare a plugin's address in `rat.yaml` for ratd to register on startup; it's kept for backward compatibility, but phone-home is the model.)
</Callout>

## Capabilities

A plugin's `Describe` lists the **capabilities** it provides — well-known strings ratd understands:

| Capability | What ratd does with it |
|------------|------------------------|
| `auth` | Calls the plugin's `Authenticate` RPC from the auth middleware to resolve the calling user from a bearer token |
| `enforcement` | Calls the plugin's `Authorize` RPC to allow/deny an action on a resource |
| `executor` | Routes pipeline-run execution to the plugin instead of the built-in warm-pool runner |
| `sharing` | Routes resource grant/revoke operations to the plugin |
| `cloud` | Asks the plugin to vend short-lived credentials for external services |

With **no** plugin for a capability, the core default applies: every request is one anonymous user (`auth`), everything is allowed (`enforcement`), runs go to the built-in runner (`executor`), there's nothing to share (`sharing`), and only local object storage is available (`cloud`). That's what keeps RAT fully usable on its own.

Capabilities are just *declarations* — a single plugin can provide several, and what each one actually does (which identity provider, which ACL model, which compute backend) lives entirely inside the plugin.

## HTTP Routes & the Reverse Proxy

A plugin can expose its own HTTP API. It declares its routes in `Describe`, and ratd publishes them under a per-plugin prefix:

```
/api/v1/x/<plugin-name>/...   →   <plugin-addr>/...
```

ratd is the **only** way in. The reverse proxy ([`plugin_proxy.go`](https://github.com/squat-collective/rat/blob/main/platform/internal/api/plugin_proxy.go)):

- strips the `/api/v1/x/<name>` prefix before forwarding;
- injects `X-RAT-Plugin-Token` (the per-startup token from `Describe`) so the plugin can tell "request via ratd" from a direct peer call on the docker network, and reject the latter;
- forwards `X-Request-ID` (tracing) and the authenticated `X-Forwarded-User`;
- **sanitizes response headers** — plugins are untrusted, so everything outside a small safe whitelist is stripped (a plugin can't set `Set-Cookie`, override the portal's `Content-Security-Policy`, or return an off-domain `Location`).

---

## Events

ratd runs an internal event bus. A plugin subscribes to channels via `Describe.event_subscriptions`, and ratd delivers each matching event to the plugin's `HandleEvent` RPC (with a short timeout, fire-and-forget).

Well-known channels:

`run_completed` · `pipeline_created` · `pipeline_updated` · `pipeline_published` · `pipeline_deleted` · `file_uploaded` · `quality_failed` · `schedule_fired`

This is how a plugin reacts to things happening in the platform — sending a notification when a run finishes, recomputing a view when a pipeline changes, and so on — without polling.

## Configuration

Plugins are configured through RAT, not through their own env soup. The flow is **pull, not push**:

<Steps>

### Declare a schema

The plugin advertises a `config_schema_json` (JSON Schema) in `Describe`.

### Edit in the portal

The portal renders a settings form straight from that schema. Saving it calls `PUT /api/v1/plugins/<name>/config`, and ratd stores the config in `plugin_catalog`.

### Plugin polls for its config

ratd **stores** config but never pushes it. The plugin polls `GET /api/v1/plugins/<name>` for its own entry and applies changes live.

</Steps>

### Management API

ratd exposes a small public API for managing the registry (used by the portal):

| Method | Endpoint | Purpose |
|--------|----------|---------|
| `GET` | `/api/v1/plugins` | List registered plugins + health |
| `GET` | `/api/v1/plugins/{name}` | One plugin's record (incl. its config — this is what plugins poll) |
| `PUT` | `/api/v1/plugins/{name}/config` | Update a plugin's config |
| `PUT` | `/api/v1/plugins/{name}/enable` · `/disable` | Toggle a plugin without removing it |
| `DELETE` | `/api/v1/plugins/{name}` | Remove a plugin from the registry |
| `GET` | `/api/v1/plugins/{name}/ui/bundle.js` | Serve the plugin's portal UI bundle |

## Health Checking

ratd health-checks every registered plugin on a **30-second** loop (5-second timeout per call) via the `HealthCheck` RPC. The result is reflected in the plugin's status in `plugin_catalog` and surfaced through `/api/v1/plugins` and the fleet metrics (`ratd_plugins_total` / `ratd_plugins_healthy`). An unhealthy plugin is left registered but not proxied to, and the relevant capability falls back to its core default.

## Portal UI (Layer 3)

A platform plugin can ship its own front-end. It declares a `ui` block in `Describe`:

- `bundle_url` — the plugin's JavaScript bundle (served back through ratd at `/api/v1/plugins/<name>/ui/bundle.js`)
- `nav_items` — entries to add to the portal sidebar
- `slots` — named UI slots in the portal the plugin fills (e.g. a pipeline-editor side panel, a table-detail action)
- `routes` — client-side routes the plugin owns, mounted under `/x/<name>`
- `bundle_hash` — an SRI hash so the browser rejects a tampered bundle

The portal loads the bundle and the plugin registers itself in the browser:

```js
window.__RAT_REGISTER_PLUGIN("my-plugin", {
  navItems: [{ label: "My Plugin", icon: "box", path: "/x/my-plugin" }],
  routes:   [{ path: "/x/my-plugin", component: "MyPluginPage" }],
})
```

So a plugin can add a whole new section to the web IDE — its own API, its own pages, its own widgets — without any change to core portal code.

## Runner Plugins (Layer 1)

Runner plugins extend the **data processing** layer. They are ordinary Python packages discovered via `importlib.metadata.entry_points()` — the same mechanism pip, pytest, and others use — and are scanned at the **start of each pipeline run** rather than at server startup.

| Extension point | Entry-point group | Protocol | Adds |
|-----------------|-------------------|----------|------|
| Merge strategies | `rat.strategies` | `MergeStrategyProtocol` | New ways to write data into Iceberg |
| Hooks | `rat.hooks` | `HookProtocol` | Callbacks at phase boundaries (logging, metrics, alerts) |
| Jinja helpers | `rat.jinja_helpers` | `JinjaHelperProtocol` | Template functions for SQL pipelines |
| Pipeline types | `rat.pipeline_types` | `PipelineTypeProtocol` | Whole new pipeline languages |
| Sources | `rat.sources` | `SourceConnectorProtocol` | External data-source connectors |

Each extension point is a `Protocol` (structural typing — no base class to inherit). Install a package into the runner image and it's discovered automatically; an invalid plugin is logged and skipped, never breaking the run. ratd surfaces the discovered set read-only at `GET /api/v1/runner/plugins`.

A single platform plugin commonly spans **Layer 2 and Layer 3** in one container — a gRPC service plus a UI bundle. Layer 1 lives entirely inside the runner and is independent of the other two.

## Writing a Plugin

The contracts are open — anything that implements them is a valid plugin.

- **Runner plugin:** pick an extension point, implement its `Protocol`, declare it under the right `entry_points` group in `pyproject.toml`, and `pip install`. RAT discovers it on the next run. (Protocol signatures and a quick-start are in the [plugins guide](/guides/writing-plugins).)
- **Platform plugin:** implement the base `PluginService` (`HealthCheck` + `Describe`, plus whatever capabilities/routes/events you advertise), phone home to ratd on startup, and package it as a container. The Go SDK provides the plumbing — phone-home loop, standard route mounting, and a `Describe` builder — so a plugin is mostly your own logic plus a few lines of registration.

The example plugins under `plugins/` are the canonical reference — copy the closest one as a starting point.

<Callout type="warning">
A platform plugin is untrusted from ratd's point of view: its responses are header-sanitized, its inbound proxy requests carry a platform token it should verify, and a failed health check takes it out of rotation. Build accordingly.
</Callout>
