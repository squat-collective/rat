# Plugin UI Development

This guide explains how a plugin adds UI to the RAT portal. UI is **Layer 3** of the [plugin system](/contributing/architecture/plugin-system): a platform plugin ships a JavaScript bundle, ratd serves it, and the portal loads it **at runtime** — there is no build-time configuration and no portal rebuild.

## How It Works

A platform plugin (a self-registering container) declares a UI in its `Describe` response. The portal then discovers and loads it on every page load:

<Steps>

### The plugin declares its UI

In its `Describe` response, the plugin sets a `ui` descriptor: `bundle_url`, an optional `bundle_hash` (SRI), and the `nav_items`, `slots`, and client `routes` it contributes. ratd records this in the plugin registry and serves the bundle back through its own origin at `GET /api/v1/plugins/{name}/ui/bundle.js`.

### The portal fetches enabled plugins

On load, the portal calls `GET /api/v1/plugins?status=enabled` and keeps the ones whose `descriptor.ui.bundle_url` is set.

### The portal injects each bundle

For each such plugin it injects a `<script>` tag pointing at `/api/v1/plugins/{name}/ui/bundle.js`. If the plugin advertised a `bundle_hash`, the tag gets a Subresource-Integrity `integrity` attribute so the browser refuses a tampered bundle.

### The bundle registers itself

As it executes, the bundle calls the global hook the loader set up:

```js
window.__RAT_REGISTER_PLUGIN("my-plugin", {
  navItems: [{ label: "My Plugin", icon: "box", href: "/x/my-plugin", priority: 50 }],
  routes:   [{ path: "/x/my-plugin", component: MyPluginPage }],
  slots:    { "sidebar-nav-extra": [MyWidget] },
})
```

The loader merges every plugin's registration into one `PluginRegistry` (`{ slots, navItems, routes }`). `<PluginSlot>` renders slot components, nav items appear in the sidebar, and routes mount under `/x/<name>`.

</Steps>

<Callout type="warning">
**Always use the two-argument form** — `window.__RAT_REGISTER_PLUGIN(name, registration)`. Passing a single object crashes the loader (`Cannot read properties of undefined (reading 'slots')`).
</Callout>

## Authoring the Bundle

A plugin UI bundle is plain JavaScript that runs in the portal's page — it is **build-free**: hand-written `React.createElement` calls, no JSX and no bundler step, so a plugin doesn't need a Node toolchain.

```js filename="bundle.js (served by the plugin, build-free)"
(function () {
  const e = React.createElement;

  function MyPluginPage() {
    return e("div", { className: "brutal-card p-4" },
      e("h1", { className: "neon-text" }, "My Plugin"));
  }

  function MyWidget() {
    return e("div", { className: "border-t border-border/50 px-3 py-2" },
      e("span", { className: "text-[10px] font-bold tracking-wider text-muted-foreground" }, "MY PLUGIN"));
  }

  window.__RAT_REGISTER_PLUGIN("my-plugin", {
    navItems: [{ label: "My Plugin", icon: "box", href: "/x/my-plugin", priority: 50 }],
    routes:   [{ path: "/x/my-plugin", component: MyPluginPage }],
    slots:    { "sidebar-nav-extra": [MyWidget] },
  });
})();
```

`React` is provided on `window` by the portal — don't bundle your own copy.

### Registration shape

| Field | Type | Purpose |
|-------|------|---------|
| `navItems` | `{ label, icon, href, priority }[]` | Sidebar entries (icon = a lucide icon name; lower `priority` sorts first) |
| `routes` | `{ path, component }[]` | Full-page client routes, mounted under `/x/<name>` |
| `slots` | `Record<string, Component[]>` | Components injected into named portal slots |

See the [Plugin Slots Reference](/reference/plugin-slots) for the full list of slot names, their locations, and props.

### Declaring the UI in `Describe`

The portal only loads a bundle the plugin advertised. In the plugin's `Describe` response, populate the `ui` descriptor:

- `bundle_url` — where the bundle lives (ratd re-serves it at `/api/v1/plugins/{name}/ui/bundle.js`)
- `bundle_hash` — SHA-256 of the bundle in SRI format (`sha256-<base64>`)
- `nav_items`, `slots`, `routes` — optional metadata mirroring what the bundle registers

<Callout type="info">
**SRI is on by default.** A bundle without a `bundle_hash` is **refused** (no `<script>` injected). For local development only, set `NEXT_PUBLIC_RAT_ALLOW_UNSIGNED_BUNDLES=true` on the portal to downgrade that to a warning. Never enable it in production.
</Callout>

## Design System Guidelines

Plugin components render inside the portal's layout. Follow the portal theme:

- **Use existing Tailwind classes** — `brutal-card`, `neon-text`, `text-[10px]`, `tracking-wider`
- **No rounded corners** — `--radius: 0px` is the portal theme
- **Font**: JetBrains Mono (inherited from `body`)
- **Colors**: use CSS variables — `text-primary`, `text-muted-foreground`, `border-border/50`
