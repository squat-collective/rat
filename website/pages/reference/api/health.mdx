# Health

Health endpoints provide liveness checks, readiness checks, version information, Prometheus metrics, and feature detection. These endpoints are mounted at the root level (outside `/api/v1`) and do not require authentication.

---

## Endpoints

| Method | Endpoint | Description |
|---|---|---|
| `GET` | `/health` | Liveness check (always 200) |
| `GET` | `/health/live` | Version and build info |
| `GET` | `/health/ready` | Dependency health checks |
| `GET` | `/metrics` | Prometheus metrics |
| `GET` | `/api/v1/features` | Active platform features and plugins |

---

## Liveness Check

```
GET /health
```

Always returns `200 OK` if the process is running. Used by Docker health checks and load balancers to determine if the container is alive.

### Request

```bash
curl http://localhost:8080/health
```

### Response — `200 OK`

```json
{
  "status": "ok",
  "version": "0.2.0",
  "git_commit": "a1b2c3d",
  "build_time": "2026-02-15T10:00:00Z",
  "go_version": "go1.24.0"
}
```

<Callout type="info">
This endpoint is intentionally simple and has no dependencies. If the HTTP server can respond, the service is alive (`status` is always `"ok"`). It also reports build/version metadata. For dependency-aware health checking, use `GET /health/ready`.
</Callout>

## Version Info

```
GET /health/live
```

Returns version and build information for the running ratd instance.

### Request

```bash
curl http://localhost:8080/health/live
```

### Response — `200 OK`

```json
{
  "status": "ok",
  "version": "0.2.0",
  "go_version": "go1.24.0",
  "build_time": "2026-02-15T10:00:00Z",
  "git_commit": "a1b2c3d"
}
```

<Callout type="info">
`/health/live` and `/health` (the backward-compatible alias) return the same payload. Version, commit, and build time are injected at release build time; local/unstamped builds report `"dev"` / `"unknown"`.
</Callout>

### Response Fields

| Field | Type | Description |
|---|---|---|
| `status` | `string` | Always `ok` |
| `version` | `string` | RAT platform version (semver; `dev` for local builds) |
| `go_version` | `string` | Go compiler version |
| `build_time` | `string` | ISO 8601 build timestamp |
| `git_commit` | `string` | Short git commit hash |

## Readiness Check

```
GET /health/ready
```

Checks the health of all configured dependencies (Postgres, S3, runner, query service). Each dependency is checked concurrently with a 2-second timeout. Returns `200 OK` only when every check passes. Returns `503 Service Unavailable` if any dependency is down.

### Request

```bash
curl http://localhost:8080/health/ready
```

### Response — `200 OK` (all healthy)

```json
{
  "status": "ready",
  "checks": {
    "postgres": { "status": "ok" },
    "s3": { "status": "ok" },
    "runner": { "status": "ok" },
    "query": { "status": "ok" }
  }
}
```

### Response — `503 Service Unavailable` (dependency down)

```json
{
  "status": "not_ready",
  "checks": {
    "postgres": { "status": "ok" },
    "s3": { "status": "ok" },
    "runner": { "status": "error", "error": "dial tcp: connection refused" },
    "query": { "status": "ok" }
  }
}
```

### Response Fields

| Field | Type | Description |
|---|---|---|
| `status` | `string` | Overall status: `ready` or `not_ready` |
| `checks` | `object` | Map of dependency name → check result. Only configured dependencies appear |
| `checks.<dep>.status` | `string` | Per-dependency status: `ok` or `error` |
| `checks.<dep>.error` | `string` | Error message, present only when `status` is `error` |

<Callout type="info">
Dependency keys are `postgres`, `s3`, `runner`, and `query`. A dev server with no dependencies configured returns an empty `checks` map and is still `ready`.
</Callout>

### Dependency Health

| Dependency | Check | Impact When Down |
|---|---|---|
| `postgres` | Connection ping | All API operations fail (state storage) |
| `s3` | Bucket head request | File operations and pipeline execution fail |
| `runner` | gRPC health check | Pipeline runs and previews fail |
| `query` | gRPC health check | Interactive queries and schema browsing fail |

<Callout type="info">
Kubernetes deployments should use `/health` for the liveness probe and `/health/ready` for the readiness probe. This ensures traffic is only routed to ratd instances that can serve requests with all dependencies available.
</Callout>

## Prometheus Metrics

```
GET /metrics
```

Exposes Prometheus-compatible metrics for monitoring and alerting.

### Request

```bash
curl http://localhost:8080/metrics
```

### Response — `200 OK`

Returns metrics in Prometheus text exposition format. `ratd` exposes a lightweight, hand-rolled metric set (no `prometheus/client_golang` histograms — there are no per-request latency or HTTP counters):

```
# HELP ratd_info Build information about ratd.
# TYPE ratd_info gauge
ratd_info{version="0.2.0",git_commit="a1b2c3d",go_version="go1.24.0"} 1

# HELP ratd_goroutines Number of goroutines.
# TYPE ratd_goroutines gauge
ratd_goroutines 42

# HELP ratd_plugins_total Total number of registered plugins (any status).
# TYPE ratd_plugins_total gauge
ratd_plugins_total 3
```

### Exposed Metrics

| Metric | Type | Description |
|---|---|---|
| `ratd_info` | gauge | Build info (labels `version`, `git_commit`, `go_version`); value always `1` |
| `ratd_goroutines` | gauge | Number of goroutines |
| `ratd_memory_alloc_bytes` | gauge | Current heap allocation in bytes |
| `ratd_memory_sys_bytes` | gauge | Total memory obtained from the OS in bytes |
| `ratd_gc_completed_total` | counter | Completed GC cycles |
| `ratd_sse_connections_active` | gauge | Active SSE connections (when the SSE limiter is enabled) |
| `ratd_postgres_pool_total` | gauge | Configured size of the main Postgres pool |
| `ratd_postgres_pool_acquired` | gauge | Currently-acquired connections from the main pool |
| `ratd_postgres_heartbeat_pool_total` | gauge | Configured size of the dedicated heartbeat pool (when enabled) |
| `ratd_postgres_heartbeat_pool_acquired` | gauge | Currently-acquired connections from the heartbeat pool |
| `ratd_plugins_total` | gauge | Registered plugins (any status) |
| `ratd_plugins_healthy` | gauge | Plugins currently enabled (passing health-loop probes) |
| `ratd_scheduler_last_tick_duration_seconds` | gauge | Duration of the most recent scheduler tick |
| `ratd_scheduler_last_tick_dispatched_total` | gauge | Schedules dispatched in the most recent scheduler tick |

Pool, heartbeat-pool, SSE, plugin, and scheduler metrics are only emitted when the corresponding subsystem is active.

<Callout type="info">
Configure your Prometheus instance to scrape `http://ratd:8080/metrics` at your desired interval. The metrics endpoint is unauthenticated and does not require API keys.
</Callout>

## Platform Features

```
GET /api/v1/features
```

Returns the active platform capabilities and loaded plugins. The portal uses this endpoint to show or hide UI elements based on available features.

<Callout type="info">
Unlike other health endpoints, this endpoint is under `/api/v1` and respects the auth middleware when authentication is enabled.
</Callout>

### Request

```bash
curl http://localhost:8080/api/v1/features
```

### Response — `200 OK` (default, no optional plugins)

```json
{
  "edition": "community",
  "namespaces": false,
  "multi_user": false,
  "landing_zones": true,
  "plugins": {}
}
```

### Response — `200 OK` (with optional plugins enabled)

```json
{
  "edition": "community",
  "namespaces": true,
  "multi_user": true,
  "landing_zones": true,
  "plugins": {
    "auth": { "enabled": true, "type": "keycloak" },
    "sharing": { "enabled": true },
    "executor": { "enabled": true, "type": "warmpool" }
  }
}
```

### Response Fields

| Field | Type | Description |
|---|---|---|
| `edition` | `string` | Build edition identifier returned by ratd (currently `community`) |
| `plugins` | `object` | Status of each active plugin, keyed by plugin/slot name |
| `namespaces` | `boolean` | Whether multi-namespace support is active |
| `multi_user` | `boolean` | Whether multi-user mode is active (provided by an auth plugin) |
| `landing_zones` | `boolean` | Whether landing zones are available |

<Callout type="info">
The exact shape of `plugins` depends on which optional plugins you have installed. Features such as multi-user, sharing, and a container executor are all delivered as free optional plugins.
</Callout>
