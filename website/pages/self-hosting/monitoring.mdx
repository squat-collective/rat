# Monitoring

RAT exposes health endpoints, structured logs, and Docker health checks to help you monitor the platform. This guide covers what to monitor, how to set up alerts, and how to diagnose common issues.

---

## Health Endpoints

The `ratd` API server exposes three health endpoints:

### GET /health

Liveness check (always `200 OK`). Returns build/version metadata. This is the backward-compatible alias for `/health/live`.

```bash filename="Terminal"
curl -s http://localhost:8080/health | jq .
```

```json filename="Response"
{
  "status": "ok",
  "version": "0.2.0",
  "git_commit": "a1b2c3d",
  "build_time": "2026-02-15T10:00:00Z",
  "go_version": "go1.24.0"
}
```

This endpoint does **not** check dependencies — `status` is always `"ok"` while the HTTP server is up. For dependency-aware checks, use `/health/ready`.

### GET /health/live

Liveness probe — identical payload to `/health`. Returns `200 OK` if the `ratd` process is running, regardless of dependency health. Use this for container orchestrators that need to know if the process is alive.

```bash filename="Terminal"
curl -s -o /dev/null -w "%{http_code}" http://localhost:8080/health/live
# 200
```

### GET /health/ready

Readiness probe. Returns `200 OK` (status `"ready"`) when every configured dependency check passes, or `503 Service Unavailable` (status `"not_ready"`) if any is down. Each dependency is checked concurrently with a 2-second timeout.

```bash filename="Terminal"
curl -s http://localhost:8080/health/ready | jq .
```

```json filename="Response"
{
  "status": "ready",
  "checks": {
    "postgres": { "status": "ok" },
    "s3": { "status": "ok" },
    "runner": { "status": "ok" },
    "query": { "status": "ok" }
  }
}
```

Each check object has a `status` of `"ok"` or `"error"`; on failure it also carries an `error` string. The dependency keys are `postgres`, `s3`, `runner`, and `query` — and only dependencies that are actually configured appear in `checks`.

<Callout type="info">
In Kubernetes deployments, use `/health` (or `/health/live`) as the `livenessProbe` and `/health/ready` as the `readinessProbe`. In Docker Compose, the built-in health check already uses the `/ratd healthcheck` CLI command which performs a similar check.
</Callout>

## Prometheus Metrics

`ratd` exposes a Prometheus metrics endpoint:

### GET /metrics

```bash filename="Terminal"
curl -s http://localhost:8080/metrics
```

<Callout type="info">
`ratd` exposes a lightweight, hand-rolled metric set (no `prometheus/client_golang` histograms). There are **no** per-request HTTP counters or latency histograms — the metrics focus on process, pool, plugin, and scheduler health.
</Callout>

```text filename="Sample output"
# HELP ratd_info Build information about ratd.
# TYPE ratd_info gauge
ratd_info{version="0.2.0",git_commit="a1b2c3d",go_version="go1.24.0"} 1

# HELP ratd_goroutines Number of goroutines.
# TYPE ratd_goroutines gauge
ratd_goroutines 42

# HELP ratd_postgres_pool_acquired Currently-acquired connections from the main Postgres pool.
# TYPE ratd_postgres_pool_acquired gauge
ratd_postgres_pool_acquired 3

# HELP ratd_scheduler_last_tick_dispatched_total Schedules dispatched in the most recent scheduler tick.
# TYPE ratd_scheduler_last_tick_dispatched_total gauge
ratd_scheduler_last_tick_dispatched_total 1
```

### Key Metrics

| Metric | Type | Description |
|---|---|---|
| `ratd_info` | gauge | Build info (labels `version`, `git_commit`, `go_version`); always `1` |
| `ratd_goroutines` | gauge | Number of goroutines |
| `ratd_memory_alloc_bytes` | gauge | Current heap allocation in bytes |
| `ratd_memory_sys_bytes` | gauge | Total memory obtained from the OS in bytes |
| `ratd_gc_completed_total` | counter | Completed GC cycles |
| `ratd_sse_connections_active` | gauge | Active SSE connections (log streams) |
| `ratd_postgres_pool_total` / `ratd_postgres_pool_acquired` | gauge | Main Postgres pool size and in-use connections |
| `ratd_postgres_heartbeat_pool_total` / `ratd_postgres_heartbeat_pool_acquired` | gauge | Dedicated heartbeat pool size and in-use connections |
| `ratd_plugins_total` / `ratd_plugins_healthy` | gauge | Registered plugins (any status) vs currently enabled |
| `ratd_scheduler_last_tick_duration_seconds` | gauge | Duration of the most recent scheduler tick |
| `ratd_scheduler_last_tick_dispatched_total` | gauge | Schedules dispatched in the most recent scheduler tick |

Pool, heartbeat-pool, SSE, plugin, and scheduler metrics are only emitted when the corresponding subsystem is active.

### Prometheus Configuration

Add RAT to your Prometheus scrape configuration:

```yaml filename="prometheus.yml"
scrape_configs:
  - job_name: 'ratd'
    static_configs:
      - targets: ['ratd:8080']
    scrape_interval: 15s
    metrics_path: /metrics
```

## Docker Health Checks

Every RAT service registers a Docker health check. Use `docker compose ps` or `docker inspect` to view health status.

### Health Check Configuration

| Service | Method | Interval | Timeout | Retries | Start Period |
|---|---|---|---|---|---|
| **ratd** | CLI: `/ratd healthcheck` | 5s | 3s | 5 | 5s |
| **ratq** | Python gRPC channel check | 10s | 5s | 5 | 15s |
| **runner** | Python gRPC channel check | 10s | 5s | 5 | 15s |
| **portal** | `wget http://localhost:3000` | 10s | 5s | 3 | 30s |
| **postgres** | `pg_isready -U rat` | 5s | 3s | 5 | 5s |
| **minio** | `mc ready local` | 5s | 3s | 5 | 5s |
| **nessie** | HTTP `/q/health/ready` | 10s | 5s | 5 | 15s |

### Checking Health Status

```bash filename="Terminal"
# Overview of all services
docker compose -f infra/docker-compose.yml ps

# Detailed health info for a specific service
docker inspect --format='{{json .State.Health}}' ratd | jq .
```

```json filename="Health output"
{
  "Status": "healthy",
  "FailingStreak": 0,
  "Log": [
    {
      "Start": "2026-02-16T09:30:00.000Z",
      "End": "2026-02-16T09:30:00.050Z",
      "ExitCode": 0,
      "Output": "ok\n"
    }
  ]
}
```

## Logging

### Log Format

All services use the Docker `json-file` log driver with rotation:

```yaml
logging:
  driver: json-file
  options:
    max-size: "10m"
    max-file: "3"
```

Each service produces up to 30 MB of logs (3 files x 10 MB) before rotation.

### Viewing Logs

```bash filename="Terminal"
# All services
make logs

# Specific service
docker compose -f infra/docker-compose.yml logs -f ratd

# Last 100 lines
docker compose -f infra/docker-compose.yml logs --tail 100 ratd

# Since a timestamp
docker compose -f infra/docker-compose.yml logs --since "2026-02-16T09:00:00" ratd
```

### Log Levels

`ratd` uses Go's `slog` structured logging:

```json
{"time":"2026-02-16T09:30:00Z","level":"INFO","msg":"server started","addr":"0.0.0.0:8080"}
{"time":"2026-02-16T09:30:05Z","level":"INFO","msg":"scheduler tick","evaluated":5,"triggered":1}
{"time":"2026-02-16T09:30:10Z","level":"ERROR","msg":"runner unreachable","addr":"runner:50052","error":"connection refused"}
```

The runner and ratq services use Python's standard logging:

```
2026-02-16 09:30:00 INFO  [rat_runner.server] gRPC server started on port 50052
2026-02-16 09:30:05 INFO  [rat_runner.executor] Starting pipeline: ecommerce.silver.clean_orders
2026-02-16 09:30:15 ERROR [rat_runner.executor] Pipeline failed: DuckDB OOM
```

### Centralized Logging

For production, forward logs to a centralized system:

<Tabs items={['Loki (Grafana)', 'ELK Stack', 'CloudWatch']}>
<Tabs.Tab>
Use the Loki Docker driver:

```yaml filename="infra/docker-compose.override.yml"
x-logging: &default-logging
  driver: loki
  options:
    loki-url: "http://loki:3100/loki/api/v1/push"
    loki-batch-size: "400"
    labels: "service={{.Name}}"
```
</Tabs.Tab>
<Tabs.Tab>
Use Filebeat to ship container logs:

```yaml filename="filebeat.yml"
filebeat.autodiscover:
  providers:
    - type: docker
      templates:
        - condition:
            contains:
              docker.container.labels.com.docker.compose.project: "rat"
          config:
            - type: container
              paths: ["/var/lib/docker/containers/${data.docker.container.id}/*.log"]
output.elasticsearch:
  hosts: ["elasticsearch:9200"]
```
</Tabs.Tab>
<Tabs.Tab>
Use the `awslogs` driver:

```yaml filename="infra/docker-compose.override.yml"
x-logging: &default-logging
  driver: awslogs
  options:
    awslogs-region: us-east-1
    awslogs-group: /ecs/rat
    awslogs-stream-prefix: rat
```
</Tabs.Tab>

## What to Alert On

### Critical (Page immediately)

| Condition | Check | Threshold |
|---|---|---|
| API down | `GET /health/live` returns non-200 | Any failure |
| Postgres down | `GET /health/ready` → `checks.postgres.status == "error"` | 2+ consecutive failures |
| S3 down | `GET /health/ready` → `checks.s3.status == "error"` | 2+ consecutive failures |
| Runner down | `GET /health/ready` → `checks.runner.status == "error"` | 3+ consecutive failures |
| Postgres pool saturated | `ratd_postgres_pool_acquired / ratd_postgres_pool_total` | Sustained near `1.0` |
| Disk full | Host disk usage | > 90% |

### Warning (Investigate during business hours)

| Condition | Check | Threshold |
|---|---|---|
| Plugins unhealthy | `ratd_plugins_total - ratd_plugins_healthy` | > 0 sustained |
| Slow scheduler ticks | `ratd_scheduler_last_tick_duration_seconds` | > a few seconds |
| Memory growth | `ratd_memory_alloc_bytes` / `ratd_goroutines` | Sustained upward trend |
| query (ratq) down | `GET /health/ready` → `checks.query.status == "error"` | 3+ consecutive failures |
| Memory pressure | Container memory usage | > 85% of limit |

### Informational

| Condition | Check | Threshold |
|---|---|---|
| Scheduler dispatch rate | `ratd_scheduler_last_tick_dispatched_total` | Sudden change in rate |
| SSE connections | `ratd_sse_connections_active` | Unusual spikes |
| Backup age | Latest backup timestamp | > 24 hours (per your schedule) |

## Troubleshooting

### Service shows "unhealthy" in docker compose ps

1. Check the service logs: `docker compose logs <service>`
2. Check the health check output: `docker inspect --format='{{json .State.Health.Log}}' <service> | jq .`
3. The most recent health check log entry shows the exit code and output

### High memory usage on runner

The runner's DuckDB instance may be processing a large dataset:

```bash filename="Terminal"
docker stats --no-stream runner
```

If memory is consistently near the limit, increase `DUCKDB_MEMORY_LIMIT` and the container memory limit. See the [Docker Compose](/self-hosting/docker-compose) page for tuning details. You can also cap per-query work with `QUERY_TIMEOUT_SECS` (ratq) and `QUALITY_TEST_TIMEOUT_SECS` (runner).

### Logs show "connection refused" errors

This typically means a downstream service has not finished starting yet. Check the health status of the target service and wait for it to become healthy.
